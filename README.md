# Submission & Ingestion Service

Prototype of the fleet telematics ingestion pipeline described in `ARCHITECTURE.md`. Uses Temporal for durable workflow orchestration and a mock Samsara API for demo data.

## Prerequisites

- Python 3.12+
- [uv](https://docs.astral.sh/uv/getting-started/installation/)
- Docker Desktop

## Setup

```bash
uv sync
```

## Running the Demo

Open **4 terminals** in this directory.

### Terminal 1 — Start Temporal

```bash
./docker/start.sh
```

Wait ~20 seconds for Temporal to initialize. UI available at <http://localhost:8080>.

### Terminal 2 — Seed demo data

```bash
uv run seed                          # 10 vehicles, 2 with missing data (defaults)
uv run seed --vehicles 20            # 20 vehicles, 2 with missing data
uv run seed --vehicles 20 --missing 5  # 20 vehicles, 5 with missing data
```

Always drops and recreates the database. Writes `demo_vehicles.json` for the mock server to load. Re-run any time to reset state.

### Terminal 3 — Mock Samsara API

```bash
uv run mock-samsara
```

Serves the seeded vehicles with telematics data on <http://localhost:8001>. **Start or restart after seeding** to pick up the latest vehicle list.

### Terminal 4 — Temporal Worker

```bash
uv run worker
```

Registers `FleetIngestionWorkflow` and `VehicleIngestionWorkflow` on the `ingestion` task queue.

### Terminal 5 — Ingestion API

```bash
uv run api
```

REST API on <http://localhost:8000>.

## Triggering Ingestion

**Create a submission** (Samsara API key required at creation time):

```bash
curl -X POST http://localhost:8000/api/v1/submissions \
  -H "Content-Type: application/json" \
  -d '{
    "agent_id": "agent-001",
    "account_id": "acme-fleet-corp",
    "vehicle_vins": ["1DEMO000000000001", "1DEMO000000000002"],
    "samsara_api_token": "demo-token-abc123"
  }'
# Returns {"id": "<submission-id>", "status": "PENDING", ...}
```

**Or use the pre-seeded demo submission** (`demo-submission-001` is created by `uv run seed`):

```bash
curl -X POST http://localhost:8000/api/v1/submissions/demo-submission-001/ingest
```

**Poll status:**

```bash
curl http://localhost:8000/api/v1/submissions/demo-submission-001
```

Expected final state:

```json
{
  "status": "READY",
  "coverage_pct": 0.8,
  "vehicle_vins": ["1DEMO000000000001", "..."]
}
```

Coverage will be < 1.0 when some vehicles have missing data (< 80% → `PARTIAL`, ≥ 80% → `READY`).

**Watch the workflow tree in Temporal UI:** <http://localhost:8080>
- `FleetIngestionWorkflow` (parent) fans out to one `VehicleIngestionWorkflow` child per VIN.

## Auth

The Samsara API key must be provided when creating a submission — it is stored and used automatically when ingestion is triggered. The mock server validates the key against a configured allowlist.

Default demo key: `demo-token-abc123`

Add more keys via env var:
```bash
SAMSARA_MOCK_API_KEYS='["demo-token-abc123","another-key"]' uv run mock-samsara
```

## Vehicle Data

### Seeded vehicles

Vehicles are generated by `uv run seed` and stored in `demo_vehicles.json`. VINs follow the pattern `1DEMO000000000001` through `1DEMO000000000N`.

Profiles cycle across vehicles (low risk → truck → EV → repeat):

| Profile | Speed | Hard brakes | Miles/day | Idle |
|---------|-------|-------------|-----------|------|
| Low risk | avg 42 mph | ~2/month | 18 | low |
| Truck | avg 58 mph | ~8/month | 320 | high |
| EV | avg 51 mph | ~1/month | 28 | minimal, fuel=0 |

### Missing data vehicles

Vehicles seeded with `--missing N` have 3 months of telematics completely absent. These are always the last N vehicles in the list. The workflow will still complete — missing-data vehicles contribute fewer records, which reduces `coverage_pct`.

All vehicles cover 2024-01-01 → 2024-12-31 (366 days, leap year).
